# Рабочий процесс разработки

## Цель

Организовать разработку так, чтобы AI-агенты могли выполнять небольшие задачи предсказуемо, а продуктовые и архитектурные решения оставались под контролем мейнтейнера.

## Ветки

```text
main
└── production

develop
└── интеграция следующего продуктового этапа

agent/<issue>-<description>
└── одна конкретная задача
```

Обычный PR направляется в `develop`. PR `develop → main` используется для выпуска согласованного набора изменений. Hotfix создаётся от `main` и направляется непосредственно в `main`.

## Единица работы

GitHub Issue является основной единицей работы. Issue должен содержать:

- контекст;
- цель;
- acceptance criteria;
- scope и out of scope;
- связанные документы и предполагаемые файлы;
- способ проверки;
- влияние на документацию.

Roadmap задаёт направление, но не заменяет Issues.

## Роли

### Мейнтейнер

- приоритизирует backlog;
- формулирует границы задачи;
- принимает продуктовые и архитектурные решения;
- проверяет PR и выполняет merge;
- следит за синхронизацией кода и документации.

### Planning/Research Agent

- изучает код и документы;
- выявляет неизвестные данные и риски;
- предлагает декомпозицию;
- не меняет production-код без отдельной задачи.

### Developer Agent

- выполняет только scope Issue;
- сначала публикует implementation plan;
- открывает Draft PR;
- запускает проверки;
- обновляет связанную документацию.

### Reviewer Agent

- проверяет acceptance criteria и diff;
- ищет незапланированные изменения;
- проверяет SEO, responsive-поведение и доступность, когда это применимо;
- не расширяет scope PR.

## Жизненный цикл задачи

```text
Product docs / Roadmap
        ↓
GitHub Issue
        ↓
agent branch
        ↓
Draft Pull Request
        ↓
Implementation + validation
        ↓
AI review
        ↓
Maintainer review
        ↓
Merge в develop
        ↓
Release PR develop → main
```

## Правила PR

- Один PR решает одну задачу.
- PR с кодом должен ссылаться на Issue.
- Draft PR открывается в начале реализации.
- Большие решения сначала оформляются исследованием или ADR.
- Попутный рефакторинг выносится в отдельный Issue.
- Визуальные изменения сопровождаются desktop- и mobile-скриншотами.
- Изменения фактов о Луизе требуют подтверждённого источника.

## Merge

Обычные изменения сливаются в `develop` после успешного CI и review. В `main` изменения попадают только через release PR либо согласованный hotfix.

После release ветка `develop` синхронизируется с новым состоянием `main`.
