# AGENTS.md

Этот файл обязателен для AI-агентов, работающих с репозиторием Luiza Estate.

## Контекст проекта

Luiza Estate — продукт вокруг персонального бренда Луизы Алиниседовой и цифровой инфраструктуры её бизнеса в недвижимости. Сайт должен укреплять доверие, объяснять услуги и приводить качественные обращения.

Перед существенной работой прочитайте:

1. `README.md`;
2. `docs/product/README.md`;
3. связанные документы из `docs/product`;
4. `docs/development/workflow.md`;
5. Issue, к которому относится работа.

## Технологии

- Node.js 22;
- Astro 6;
- TypeScript в strict-режиме;
- обычный CSS без UI-фреймворка;
- GitHub Pages;
- GitHub Actions.


## Правила работы

- Одна задача — одна ветка — один Pull Request.
- Ветка агента создаётся от `develop` и называется `agent/<issue>-<short-description>`.
- Срочные production-исправления могут идти от `main` через `agent/hotfix-<issue>-<description>`.
- Не делать прямые push в `main` или `develop`.
- Не расширять scope задачи без явной фиксации в Issue или PR.
- Не выполнять попутный рефакторинг, не требуемый acceptance criteria.
- Не менять продуктовые формулировки, контакты, юридические сведения и факты о Луизе без подтверждённого источника.
- Гипотезы должны быть явно отмечены как гипотезы.
- Существенные продуктовые решения фиксируются в `docs/product/decisions.md`.
- Существенные архитектурные решения оформляются отдельным ADR до реализации.
- Документация обновляется в том же PR, если изменение влияет на workflow, архитектуру, поведение сайта или продуктовые договорённости.

## Перед изменением кода

Оставьте короткий implementation plan в Issue или Draft PR:

1. какие файлы будут изменены;
2. какое решение предлагается;
3. основные риски;
4. как будет проверен результат.

## Критерии готовности

Работа считается готовой, когда:

- выполнены acceptance criteria;
- scope не расширен скрыто;
- проект успешно проходит restore, build и publish;
- ручная проверка описана в PR;
- для визуальных изменений приложены desktop- и mobile-скриншоты;
- обновлена связанная документация либо явно указано, почему обновление не требуется;
- в PR перечислены риски и способ отката.

## Формат Pull Request

PR должен:

- ссылаться на Issue через `Closes #...` или `Relates to #...`;
- быть Draft до завершения реализации и проверок;
- описывать результат, изменения, проверку, документацию и риски;
- быть направлен в `develop`, кроме согласованных hotfix в `main`.

## Ограничения безопасности

- Не добавлять секреты, токены, приватные ключи и персональные данные в репозиторий.
- Не публиковать неподтверждённые юридические реквизиты, клиентские кейсы или отзывы.
- Не удалять аналитику, SEO-настройки или production-конфигурацию без явного требования задачи.
